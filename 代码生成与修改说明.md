# 代码生成与修改说明

本文档记录本次从 Python 脚本工作流改造成 Qt/C++ App 原型时新增和修改的代码内容。

## 目标

原流程是先在 Chitubox 中按材料分别切片，再用 `slice_1080p.py` 合并多个材料文件夹并生成 `run.gcode`。

本次改造后的流程为：

1. 在 App 中导入多个 STL 模型。
2. 在右侧操作栏分别设置模型的平移、旋转、等比例缩放和材料。
3. App 按材料把 STL 切片导出为 PNG 文件夹。
4. App 生成 `config.yaml`，记录各材料切片文件夹路径和打印参数。
5. App 通过命令行调用 `slice_1080p.py --config <config.yaml> --output <输出目录>`。
6. Python 脚本合并多材料切片并生成最终投影 PNG 和 `run.gcode`。

## 本次修复（最新）

在原型基础上修复了以下问题（均已编译并端到端验证通过）：

1. **界面冻结**：导出流程（切片 → 写配置 → 调 Python）原本在主线程同步执行、`waitForFinished` 阻塞，期间界面卡死。现拆分到后台 `QThread` + `SliceWorker`，日志实时回传，UI 不再冻结。
2. **每材料曝光独立**：原本所有材料共用同一组曝光参数。现切片页改为“每材料曝光参数表格”，`SliceSettings` 用 `QVector<MaterialExposure>` 保存，`ConfigWriter` 按材料分别写入，参数一路贯穿到 `run.gcode`。
3. **底层/普通电流可分别设置**：原本底层电流和普通电流共用一个控件，现已在表格中分列。
4. **二进制 STL 容错**：原本要求二进制 STL 文件大小严格等于 `84 + 50×三角形数`，带结尾 padding 的二进制 STL 会加载失败；现已容忍结尾多余字节。
5. **切片性能**：切片改为按三角面 Z 范围分桶，每层只测试可能相交的三角面，提升大模型切片速度。

## 稳定性与工程化修复

这一轮是在**真机端到端实测**中发现并修复的，不实跑发现不了：

### 1. 后台线程清理时序（会导致崩溃 / SIGABRT）

- **现象**：导出完成、或用户在导出过程中退出 App 时，`QThread` 可能在仍处于运行状态时被销毁，触发 `QThread: Destroyed while thread is still running` 并 abort（退出码 134）。
- **修复**：
  - `MainWindow::workerFinished()` 在收尾（尤其自检模式直接退出前）先对线程 `quit()` + `wait()`，确保线程事件循环结束后再销毁。
  - `MainWindow` 析构函数兜底：若退出时仍有运行中的 worker 线程，先 `quit()` + `wait()` 再析构，避免中途退出崩溃。

### 2. 缺少 Bundle Identifier

- 原 `.app` 没有 bundle id。`CMakeLists.txt` 的 `APPLE` 分支补充：
  - `MACOSX_BUNDLE_GUI_IDENTIFIER com.multimaterial.slicer`
  - `MACOSX_BUNDLE_BUNDLE_NAME`、`MACOSX_BUNDLE_BUNDLE_VERSION`、`MACOSX_BUNDLE_SHORT_VERSION_STRING`
- 正式 App 本应有唯一标识；同时这是 macOS 上各类系统集成（窗口识别、权限、签名）正常工作的前提。

## 自检模式（`--selftest`）

为了在**不依赖 GUI 鼠标点击**的情况下验证"导出按钮 → 后台线程 → Python → 完成回调"这条完整链路，新增隐藏命令行模式。

- 触发：`MultiMaterialSlicer --selftest a.stl b.stl`
- 行为（`MainWindow::runSelfTest()`）：导入模型后，自动设 2 种材料、给两种材料填入**不同的**曝光参数、把模型分别指派到材料 0/1、设输出目录为 `/tmp/selftest_out`，然后调用与界面按钮**完全相同**的 `runWorkflow()`。
- 结束：`workerFinished()` 在自检模式下打印结果并以退出码返回（成功 0 / 失败 1），供脚本化回归测试使用。
- 价值：比模拟鼠标点击更严格——它跑的就是生产代码路径，且可纳入 CI / 命令行自动化。

### STL 读取

- `src/StlMesh.h`
- `src/StlMesh.cpp`

作用：

- 读取二进制 STL。
- 读取 ASCII STL。
- 存储三角面顶点和法线。
- 计算模型包围盒。
- 导入后将模型中心移动到平台中心附近，底部放到 Z=0。

ASCII / 二进制判定逻辑：

- 先检查文件文本是否含 `facet` / `vertex` 关键字，若是则按 ASCII 解析，避免文本文件被误读成二进制。

说明：

- 示例文件 `切片示例/实体1.stl` 是二进制 STL。
- 目前没有做模型修复、法线修复、非流形检查。

### STL 切片导出

- `src/SliceExporter.h`
- `src/SliceExporter.cpp`

作用：

- 按材料把当前 STL 场景导出为多个 PNG 文件夹。
- 每个材料对应一个目录：

```text
material_0/
  1.png
  2.png
material_1/
  1.png
  2.png
```

实现方式：

1. 将模型顶点按当前平移、旋转、缩放变换到世界坐标。
2. 按每个三角面的 Z 范围把它分配到所跨越的层（分桶），这样每层只需测试可能与该层平面相交的三角面，避免逐层遍历全部三角面（对大模型显著提速）。
3. 按层高逐层生成 Z 平面。
4. 对该层桶内的每个三角面求与当前 Z 平面的交线段。
5. 将交线段投影到像素平面。
6. 使用扫描线奇偶填充生成黑白 PNG。

当前适用范围：

- 适合闭合 STL 的基础切片。
- 作为 App 工作流原型已经可用。

当前限制：

- 没有支撑生成。
- 没有岛检测。
- 没有抗锯齿。
- 没有模型布尔修复。
- 对破面、非流形 STL 的容错能力有限。

## Python 脚本修改

### 修改文件

- `slice_1080p.py`

### 修改前

原脚本特点：

- 顶部直接导入 `interface_slice` 和 PyQt5。
- 启动时会创建 Qt UI。
- 核心合并逻辑在 `sliceThread.work()` 中。
- 依赖工作目录下的 UI 文件和若干外部模块。
- 不适合被 C++ App 直接作为命令行工具调用。

### 修改后

现在脚本特点：

- 去掉启动时对旧 UI 的依赖。
- 保留多材料 PNG 合并、DP 换槽路径规划、GCode 生成逻辑。
- 改为纯命令行工具。
- 支持 `--config` 和 `--output` 参数。
- 支持中文路径下的 PNG 读取和写入。
- 增加默认参数，减少 `config.yaml` 缺字段时的崩溃概率。
- 支持 `output_width/output_height` 控制最终投影图尺寸。

命令行格式：

```bash
python3 slice_1080p.py --config /path/to/config.yaml --output /path/to/merged
```

也支持：

```bash
python3 slice_1080p.py /path/to/config.yaml /path/to/merged
```

输出：

```text
change_times=40 run_time=02:51:20 output=/path/to/merged
```

### 保留的核心逻辑

- 统计每层每个材料是否有有效像素。
- 用动态规划计算材料槽切换顺序。
- 输出合并后的投影 PNG。
- 输出 `run.gcode`。
- 保留 `alpha`、`beta`、`AMS`、`ASS`、清洗、风干、延时拍摄等配置字段。
